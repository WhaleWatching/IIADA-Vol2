Hello, Bootstrap!

开发者都对 Bootstrap 有着别样的情感。

Bootstrap 是敏捷开发、用户体验、HTML 5、Web 2.0和大数据时代的产物。

在当时，「前端」已经蜕去「做美工和网页的」印象，甚至这个崭新的概念可以和高居学术层面的「后端」分庭抗礼，「前端开发规范」被人们摆上台面，终于能照着词本唱戏而免去吐露粗俗的市侩之词。那是个极度推崇「体验」的时代，历史的必然，就像「用户」在后工业时代不再喜欢看到暴露铆钉的机器。

宏观上看，是设计在新载体上的爆发，开发者和决策层都在谈论着「拟物风格」、「扁平化」，浪潮尖端的「苹果」，有一定体系的「Google」，以及浪潮另一端的「微软」。尚未写入「历史」的设计进程，正在由不同程度的「开发者」和「设计师」推动，人群中的大多数都在迷惘：在强调「用户体验」的「创业」时代中，应该涌向何方？在代码层面，一套使用框架的代码质量是不容易被用户察觉的。然而，「用户体验」层面上的「敏捷开发框架」来了。

我们说某个框架可以敏捷开发，大多指这个框架的易用、通用和有一定高度的约定。这三个属性决定了一款框架的性质－－在「人机交互界面」层面上尤为有效。在这里，易用性和通用性从结论上决定了一个框架的热度，而约定往往容易被人群忽略。

回到我们的主题，毫无疑问，Bootstrap（类似的框架还有 Foundation）有着极佳的易用性和通用性。它给了你类似于「拖拽美梦」（该词被用来形容仅需拖拽就可完成的技术生产工具）的体验－－无需精打细算，即可拥有「规范」的产出。在一些工程师看来，Bootstrap 就像一片「合格证」，如果「作品」由内而外洋溢着「Bootstrap 味道」，「OK，设计和交互被我们搞定了」。

「Bootstrap 味道」，这不是个中立词。当工程师看到精致的「Bootstrap Components Guide」，工程师希望「产出」有着一样精致的「Bootstrap 味道」。此时，再次引出我们的观点，它影响着截然不同的「Bootstrap 味道」：「约定」被人们忽略了。

「约定」，是一套框架必不可少的部分。「约定」囊括了使用说明，设计准则和最佳实践建议。Bootstrap 有一定程度的约定：大多是使用说明，极少的设计准则和最佳实践建议。我们并非在全面批判「Bootstrap」设计的不够好，或者专业程度不高，而是「Bootstrap 开发者」与「使用 Bootstrap 的工程师」有巨大的设计分歧，很多问题交给工程师无法「敏捷的」甚至无法「正确的」解决。约定不足所带来的问题很多。

  • 文本元素不尽如意，其之一是因为 Bootstrap 基于「文档超文本」－－它使用「Normalize.css」，而工程师并非都完成了从「DIV＋CSS 设计思路」的进化－－他们习惯于使用「Reset.css」。
  • Bootstrap 有数目众多看似「不完整」的组件，例如该列表，Bootstrap 并没有给予「jumbotron」一份子列表样式定义集，实则缘于 Bootstrap 开发者放宽了「Custom design」的范围。
  • Bootstrap 组件绑定了一些粗糙的动画。这是 Bootstrap 技术上的不足－－没有「Animation system」－－约定不周。
  • 我们提到「Bootstrap 组件」是不够全面、完整的，意味着工程师为了完整的表达要付出很多精力－－这个过程也就是「吹散 Bootstrap 味道」的过程。而往往他们吹的不够干净：残留在页面上的 Bootstrap 边角料非常突兀。

没有足够能力参悟「Bootstrap 隐性约定」的工程师使用「Bootstrap」的事件，我们称之为「Bootstrap 失败」。所以，对相当一部分依赖 Bootstrap 的工程师来说，「Bootstrap changes nothing」。

我们在这里不是诉求某种感情－－譬如张贴「抵制 Bootstrap」之类带有革命意味的标语。之所以选题「Bootstrap」，缘由是这是一个「代表大多数」的例子，「Bootstrap 失败」是个广泛的现象，它存在于很多各式各样的「框架」中。展览着力于展现「Bootstrap 失败」，这种「互联网」构成中的一部分。

本展尽量避免修饰或添加额外的样式规则，因此，展馆本身也是一例「Bootstrap 失败」。

Thumbnail 是 Bootstrap 的通用容器

这是 Bootstrap Well。

Well 带有凹陷效果，是 Well 有别于 Thumbnail 的特点。不难猜测，Well 被推荐用来显示次级信息。

但设计者的想法改变不了 Well 和 Thumbnail 是救命稻草的现实，当出现「不知怎样区别、突出某些内容」时的救命稻草。

所以这两个容器可以用来无差别大块划分页面构成，以区分 background。

我们可以说，Well 和 Thumbnail 多快好省的建设了社会主义。

我们认为，无论如何，Well 和 Thumbnail 都不应该嵌套。

现实中，这两个容器嵌套的不适感却不会被工程师注意

设计者的想法大多是不同的，譬如，仅从「直角组件」这一个结论出发，会追溯到多个设计流派。

Bootstrap 没有提供太多可供工程师额外使用的容器，这使得审查到「Bootstrap 容器」不够用的工程师踏出了私有风格设计的第一步。

一些生性慵懒的工程师不太在意用户对组件功能性方面的理解，慵懒的工程师苦恼于容器的选择，他们最终决定对 Alert 下手－－Alert 相较于 Thumbnail 和 Well，最大的优势在于摆脱了单一的颜色，工程师用起来随心所欲，这是设计思路其一。

我们原本不想讨论没有设计性、纯粹工具性使用「Bootstrap」的情景，因为在这种情形下，工程师没有要求框架完成它力不可及的任务。

而事实是，我们意识到，Bootstrap 不存在用户角度纯粹的工具性使用－－即使工程师认为它是纯粹工具性的，而用户却必须考量它的设计。
